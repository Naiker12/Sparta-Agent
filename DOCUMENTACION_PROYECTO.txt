================================================================================
 1. ¿QUÉ ES SPARTA AGENT?
================================================================================

Sparta Agent es una aplicación de escritorio para orquestar agentes de IA,
construida con React + TypeScript + Vite + Electron. Proporciona una interfaz
unificada de chat, sistema de agentes real con subagentes, gestión de sesiones,
skills reutilizables, servidores MCP, canales de comunicación, memoria persistente
con grafo 3D + búsqueda semántica vectorial (ChromaDB), proyectos y proveedores
de modelos de lenguaje.

Soporta el Protocolo de Contexto de Modelo (MCP), 13 temas visuales
(8 oscuros + 5 claros), internacionalización español/inglés, iconos de marca
con detección de tema, y un bus de eventos tipado en memoria. Es un
"IDE de agentes de IA" local-first en desarrollo activo.

Estado actual (v0.9.0):
  • UI/UX muy pulida y funcional (layout, sidebar, chat, settings, views)
  • Gestión de sesiones, skills, MCP, canales, memoria y proyectos operativa
  • Sistema de proveedores con test connection real y fetch de modelos
  • Grafo 3D de memoria con Three.js interactivo + extracción automática
  • MEMORIA SEMÁNTICA VECTORIAL: ChromaDB con embeddings vía OpenAI/Ollama
  • SISTEMA DE AGENTES REAL: loop LLM→tools→subagentes con ejecución paralela
  • Vault cifrado para API keys con safeStorage de Electron
  • Capa de IA funcional: transports (Anthropic, OpenAI, Ollama), factory, gateway
    con buildHeaders()/buildBody() separados, retry exponencial, errores mapeados
  • Chat real con streaming token-a-token vía IPC (main process)
  • ThinkingBlock / ThinkingPill / PipelineTrace con datos reales del proveedor
  • ToolCallSummary con expand/collapse, DiffView para edición de archivos
  • Slash commands (/clear, /model, /memory, /reasoning, /help)
  • Interrupt-and-redirect: mensajes de corrección mientras el agente genera
  • Abortar generación (AbortSignal + botón Stop)
  • Memoria como grafo de conocimiento: extracción de entidades/hechos/relaciones
  • 32 tests unitarios con vitest cubriendo transports, embeddings y agentes
  • Interfaces separadas en src/interfaces/ para mejor práctica
  • Capa de servicios (src/services/) con arquitectura limpia
  • Editor y Terminal panels son placeholders visuales

================================================================================
 4. ARQUITECTURA
================================================================================

 4.1 Flujo de la Aplicación:

   index.html
     → src/main.tsx (React root)
       → AppShell.tsx (shell raíz)
         → TitleBar.tsx (drag region, AppMenu, tabs, settings)
         → SidebarProvider → AppSidebar.tsx + Panel Activo
         → StatusBar.tsx
         → SettingsDialog.tsx (renderizado condicional)

   Panel Activo según mainView.type:
     - chat     → ChatArea.tsx
     - editor   → EditorPanel (placeholder)
     - terminal → TerminalPanel (placeholder)
     - agents   → AgentsPanel.tsx (real: cards + ejecución de tareas)
     - sessions → SessionsView.tsx
     - skills   → SkillsView.tsx
     - mcp      → McpView.tsx
     - channels → ChannelsView.tsx
     - memory   → MemoryView.tsx

 4.2 Flujo de Estado:

   Zustand Stores ←(subscribe)→ React Components
        ↓
   types/ (interfaces compartidas)
        ↓
   persist middleware → localStorage (9 stores)
   Vault (electron safeStorage) → sparta-vault.json (API keys cifradas)

 4.3 Flujo de Datos:

   • Zustand stores son la única fuente de verdad.
   • 9 stores usan persist middleware → localStorage.
   • 3 stores son en memoria: agent, event-bus, mcp.
   • API keys: el renderer almacena solo hasVaultKey: boolean; la key real
     vive cifrada en sparta-vault.json (main process via safeStorage).
   • Los hooks son wrappers delgados alrededor de los stores.
   • Event bus pub/sub con 40 tipos de eventos SpartaEvent:

 4.4 Flujo de Chat con Memoria Semántica y Vault:

   Usuario escribe mensaje
     → ChatInput.handleSend()
       → useChatSession.sendMessage(text)
         → (si semanticMemoryEnabled) buildMemoryContext(text, k=5) inyecta
           <memoria_relevante> en system prompt
         → getProviderKey(provider) resuelve key desde vault o localStorage
         → window.sparta.sendMessage({ providerKey, vendor, ... })
           → chat.ipc.ts construye endpoint/headers/body según vendor
           → fetch(api) con retry 429/529, error mapping 401/403/etc
           → stream de vuelta vía sparta:event
       → Al completar: extractMemory → writeExtractedMemory → rebuildGraph
         → (si semanticMemoryEnabled) indexInChroma(entry) en ChromaDB

 4.5 Flujo del Sistema de Agentes:

   Usuario escribe tarea en AgentsPanel
     → useAgent.executeTask(agentId, descripción)
       → registerAgent + addTask + updateAgentStatus('running')
       → dispatch agent:started
       → runAgentTask(taskId, agentId, ...)
         LOOP (max 10 turnos):
           buildPrompt(system + tools definitions + historial)
           llmCall(prompt) → respuesta
           parseToolUseBlocks() → [tool_use, ...]
           detectSubagentRequest() → delegar? spawnSubagent()
           executeToolsParallel() si ≥ 2 tools independientes
           feed results al LLM
           updateTask steps en vivo
       → updateAgentStatus('completed')
       → dispatch agent:completed

 8. CAPA DE TRANSPORTES (doc 01)
================================================================================

   Cada transport (Anthropic, OpenAI, Ollama) implementa:

     buildHeaders(): Record<string, string>
       → Anthropic: x-api-key, anthropic-version, content-type
       → OpenAI: Authorization Bearer, content-type
       → Ollama: content-type (sin auth)

     buildBody(req): Record<string, unknown>
       → Cada proveedor construye su body específico

   http-utils.ts (compartido):
     HTTP_STATUS_MESSAGES: mapeo 400→"Solicitud inválida", 401→"API key
       inválida o expirada", 429→"Rate limit del proveedor", 529→
       "Proveedor sobrecargado"
     isRetryable(status): true para 429 y 529
     fetchWithRetry(url, options, retries=1): reintento con backoff
       exponencial (1s, 2s) para códigos retryable

   chat.ipc.ts:
     Recibe vendor explícito en lugar de adivinar por subcadena en URL
     Endpoint correcto para Anthropic: /v1/messages (no /v1/chat/completions)
     Error mapping con HTTP_STATUS_MESSAGES
     Retry 429/529 con backoff exponencial

 9. MEMORIA SEMÁNTICA VECTORIAL (doc 02)

   chroma-client.ts:

     Después de extraer memoria:
       → indexInChroma(entry) para cada nueva entrada

 10. SISTEMA DE AGENTES (doc 03)
================================================================================

   agent-runtime.ts:
     runAgentTask(taskId, agentId, description, systemPrompt, model,
       allowedTools, toolDefinitions, toolRunner, llmCall, depth, tokenBudget)
     LOOP (max 10 turnos):
       buildPrompt() → arma system + tools definitions + historial
       llmCall(prompt) → respuesta del LLM
       parseToolUseBlocks() → extrae <tool_use> del texto
       detectSubagentRequest() → extrae <delegar><tarea> del texto
       Si hay subagent task → spawnSubagent()
       Si hay tool_use → executeTool o executeToolsParallel
       Feed resultados al LLM
     Emite eventos agent:started, agent:completed, tool:called, tool:result, tool:error

   tool-executor.ts:
     executeTool(toolCall, runner): ejecuta tool, mide timing, emite eventos
     executeToolsParallel(toolCalls, runner): Promise.all para calls independientes
     areToolCallsIndependent(calls): true si hay ≥ 2 calls
     buildToolDefinitions(mcpTools): convierte MCPTool[] a formato LLM tool definitions

   subagent-spawner.ts:
     spawnSubagent(description, options, model, llmCall, toolDefs, toolRunner)
     Límites: MAX_DEPTH=2, TASK_TIMEOUT_MS=120s, MAX_TOKENS_PER_TASK=32000
     Crea agente hijo en store, emite eventos, corre con Promise.race(timeout)

   task-planner.ts:
     planTask(description, availableTools): genera 4 pasos típicos

   useAgent.ts (hook):
     executeTask(agentId, taskDescription, systemPrompt?): orquesta runtime
     createAgent(data): registra nuevo agente
     Expone agents, tasks, activeTasks

 11. VAULT DE SEGURIDAD (doc 05 — 5.1)
================================================================================

   electron/vault.ts:
     safeStorage.encryptString() / decryptString() para cifrar API keys
     Persistencia en sparta-vault.json (en APP_ROOT)
     storeKey(keyId, value, vendor): cifra y guarda
     getKey(keyId): descifra y devuelve
     deleteKey(keyId): elimina del vault
     listKeys(): lista keyIds sin valores
     hasKey(keyId): boolean

   electron/ipc/vault.ipc.ts:
     6 handlers IPC: vault:isAvailable, vault:storeKey, vault:getKey,
       vault:deleteKey, vault:listKeys, vault:hasKey

   src/lib/vault-helper.ts:
     getProviderKey(provider): resuelve key desde vault o apiKey legacy
     storeInVault(providerId, apiKey, vendor): store via IPC
     removeFromVault(providerId): delete via IPC
     isVaultAvailable(): chequea disponibilidad de safeStorage

   Flujo de seguridad:
     1. Usuario ingresa key en ConfigureProviderDialog
     2. handleSave → storeInVault(id, key) → IPC → main process
     3. main: safeStorage.encryptString(key) → sparta-vault.json
     4. provider.store: apiKey=undefined, hasVaultKey=true
     5. Renderer NUNCA ve la key (excepto momento de input)
     6. Al enviar: getProviderKey(provider) → vault IPC → descifra → fetch
