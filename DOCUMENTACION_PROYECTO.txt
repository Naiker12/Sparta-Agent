================================================================================
 1. ¿QUÉ ES SPARTA AGENT?
================================================================================

Sparta Agent es una aplicación de escritorio para orquestar agentes de IA,
construida con React + TypeScript + Vite + Electron. Proporciona una interfaz
unificada de chat, sistema de agentes real con subagentes, gestión de sesiones,
skills reutilizables, servidores MCP, canales de comunicación, memoria persistente
con grafo 3D + búsqueda semántica vectorial (ChromaDB), proyectos y proveedores
de modelos de lenguaje.

Soporta el Protocolo de Contexto de Modelo (MCP), 13 temas visuales
(8 oscuros + 5 claros), internacionalización español/inglés, iconos de marca
con detección de tema, y un bus de eventos tipado en memoria. Es un
"IDE de agentes de IA" local-first en desarrollo activo.

Estado actual (v0.2.0):
  • UI/UX muy pulida y funcional (layout, sidebar, chat, settings, views, todos los diálogos con shadcn/inline overlay)
  • Gestión de sesiones, skills, MCP, canales, memoria y proyectos operativa
  • Sistema de proveedores con test connection real y fetch de modelos
  • Grafo 2D de memoria con D3 Force + Canvas (reemplaza Three.js) + extracción automática
  • MEMORIA SEMÁNTICA VECTORIAL: ChromaDB con embeddings vía OpenAI/Ollama
  • SISTEMA DE AGENTES REAL: loop LLM→tools→subagentes con ejecución paralela
  • Vault cifrado para API keys con safeStorage de Electron
  • Capa de IA funcional: transports (Anthropic, OpenAI, Ollama), factory, gateway
  • ThinkingBlock / ThinkingPill / PipelineTrace con datos reales del proveedor
  • ToolCallSummary con expand/collapse, DiffView para edición de archivos
  • Slash commands (/clear, /model, /memory, /reasoning, /help)
  • Interrupt-and-redirect + Abort generation (AbortSignal + botón Stop)
  • Memoria como grafo de conocimiento: extracción LLM de entidades/hechos
  • 59 tests JS (vitest) + 32 tests Python (pytest) + 9 tests Rust (cargo)
  • PYTHON SIDECAR: sparta_ai/ con LangGraph, subagentes, ChromaDB VectorStore
  • RUST SECURITY LAYER: napi-rs native addon (validator/sanitizer/guard/audit)
  • CHAT IPC DELEGADO AL SIDECAR: chat.ipc.ts envía vía stdio JSON-RPC
  • FIX THINKING FROZEN: thinkingStatus como estado explícito, auto-collapse
  • TERMINAL CON PTY REAL: node-pty + PowerShell/bash con resize sincronizado
  • Integración agente ↔ terminal: el agente de IA puede ejecutar comandos
  • Tema dinámico: colores de xterm siguen el tema activo de Sparta
  • Lazy loading de vistas pesadas (Suspense + React.lazy)
  • Stores con persist: mcp.store.ts (antes en memoria), settings migrate v3
  • Cron jobs, gateway monitor, usage tracker operativos
  • sidecar-bridge.ts: puente simplificado desde el frontend al sidecar Python
  • EDITOR FUNCIONAL: Monaco Editor con 9 lenguajes, toolbar, selector sintaxis
  • TYPES CONSOLIDADOS: src/interfaces/ eliminado — todo en src/types/
  • D3 FORCE GRAPH: MemoryGraphScene reemplazado por MemoryGraphD3 (Canvas 2D, ~45KB vs ~580KB Three.js)
  • SEARCH IPC DEPRECADO: search.ipc.ts reemplazado por web_search tool del sidecar Python
  • PYTHON EMBEDDED: electron-builder configurado para empaquetar runtime Python
  • SIDECAR CON PYTHON PORTABLE: getPythonCommand() detecta bundled o system Python
  • TERMINAL FIX: fondo oscuro forzado (siempre #0C0C10, sin importar el tema UI)
  • TERMINAL BANNER: ASCII art Sparta al crear sesión + buffer de datos tempranos para evitar prompt perdido
  • TERMINAL AGENT PREFIX: comandos del agente con prefijo [agente] visible en terminal
  • EDITOR DELETE: eliminar archivos/carpetas vía shell.trashItem() (papelera del SO)
  • EDITOR EXPLORER TOGGLE: Ctrl+B ocuestra/muestra el panel de explorador de archivos
  • CONFIRM DELETE DIALOG: reutilizado para todas las operaciones de borrado (shadcn Dialog)
  • MODAL.TSX ELIMINADO: todos los diálogos migrados a shadcn Dialog o overlay inline
  • SKILLS TAB: Configuración ahora lista skills del sistema + skills de usuario con buscador
  • TOKEN TRACKING REAL: currentTurnInput/currentTurnOutput en usage.store, sin datos hardcodeados

  4.1 Flujo de la Aplicación:

   index.html
     → src/main.tsx (React root)
       → AppShell.tsx (shell raíz)
         → TitleBar.tsx (drag region, AppMenu, tabs, settings)
         → SidebarProvider → AppSidebar.tsx + Panel Activo
         → StatusBar.tsx
         → SettingsDialog.tsx (renderizado condicional)

   Panel Activo según mainView.type:
      - chat     → ChatArea.tsx
      - editor   → EditorPanel.tsx (Monaco Editor funcional con 9 lenguajes)
       - terminal → TerminalPanel.tsx (PTY REAL: xterm.js + node-pty + PowerShell/bash)
      - agents   → AgentsPanel.tsx (real: cards + ejecución de tareas)
      - sessions → SessionsView.tsx (lazy loaded)
      - skills   → SkillsView.tsx (lazy loaded)
      - mcp      → McpView.tsx (lazy loaded)
      - channels → ChannelsView.tsx (lazy loaded)
      - memory   → MemoryView.tsx (lazy loaded)

  4.2 Flujo de Estado:

   Zustand Stores ←(subscribe)→ React Components
        ↓
   types/ (definiciones inline, sin src/interfaces/)
        ↓
   persist middleware → localStorage (11 stores)
   Vault (electron safeStorage) → sparta-vault.json (API keys cifradas)

 4.3 Flujo de Datos:

    • Zustand stores son la única fuente de verdad.
    • 11 stores usan persist middleware → localStorage.
    • 4 stores son en memoria: agent, event-bus, mcp, gateway.
   • API keys: el renderer almacena solo hasVaultKey: boolean; la key real
     vive cifrada en sparta-vault.json (main process via safeStorage).
   • Los hooks son wrappers delgados alrededor de los stores.
    • Event bus pub/sub con 40 tipos de eventos SpartaEvent:
        agent:started, agent:completed, agent:error
        thinking:started, thinking:token, thinking:completed
        tool:called, tool:result, tool:error
        mcp:connected, mcp:tool_discovered, mcp:disconnected
        stream:token, stream:completed, stream:error
        pipeline:step
        chat:message, message:deleted, message:edited, message:shared
        session:created, session:switched
        project:created, project:switched, project:deleted
        skill:created, skill:updated, skill:deleted, skill:invoked
        channel:created, channel:message, channel:deleted
        memory:added, memory:updated, memory:deleted
        memory:semantic_search, memory:indexed, memory:extraction_empty

  4.4 Flujo de Chat con Sidecar Python + Rust Security:

   Usuario escribe mensaje
     → ChatInput.handleSend()
       → useChatSession.sendMessage(text)
         → getProviderKey(provider) resuelve key desde vault o localStorage
         → window.sparta.sendMessage({ providerKey, vendor, mode, skills, ... })
            → Electron main process recibe chat:send IPC
            → Rust Security Layer: valida esquema del mensaje (validator.rs)
            → keymanager.ipc.ts: asegura que el sidecar tenga las keys en caché
            → sidecar.ipc.ts: getPythonCommand() detecta bundled (python-runtime) o system Python; envía JSON-RPC a Python via stdin/stdout
             → Python sparta_ai/server.py: dispatch a chat.stream
               → sparta_agent.py: LangGraph StateGraph con router de intención
               → streaming vía astream_events → event_bridge.py → stdout
           → chat.ipc.ts: lee eventos de stdout, los reenvía al renderer
         → useChatSession.onEvent: recibe thinking:started/token/completed, stream:token/completed
         → chat.store: onThinkingStart/Token/End, onStreamEnd actualizan thinkingStatus
       → Al completar: extractMemory → writeExtractedMemory → rebuildGraph
         → (si semanticMemoryEnabled) indexInChroma(entry) en ChromaDB

   FLUJO DE SEGURIDAD RUST:
     ↑ Electron envía mensaje → Rust valida esquema (validator.rs)
     ↑ Tool call del agente → Rust sanitiza paths/URLs (sanitizer.rs)
     ↑ Cada minuto → Rust verifica rate limit por sesión (guard.rs)
     ↑ Eventos de seguridad → Rust persiste audit log (audit.rs)

 4.5 Flujo del Sistema de Agentes:

   Usuario escribe tarea en AgentsPanel
     → useAgent.executeTask(agentId, descripción)
       → registerAgent + addTask + updateAgentStatus('running')
       → dispatch agent:started
       → runAgentTask(taskId, agentId, ...)
         LOOP (max 10 turnos):
           buildPrompt(system + tools definitions + historial)
           llmCall(prompt) → respuesta
           parseToolUseBlocks() → [tool_use, ...]
           detectSubagentRequest() → delegar? spawnSubagent()
           executeToolsParallel() si ≥ 2 tools independientes
           feed results al LLM
           updateTask steps en vivo
       → updateAgentStatus('completed')
       → dispatch agent:completed

 4.6 Integración Electron:

   • vite-plugin-electron/simple compila:
       electron/main.ts → dist-electron/main.js
       electron/preload.ts → dist-electron/preload.mjs
   • En dev: carga desde Vite dev server URL
      • En prod: carga dist/index.html
      • Ventana: 1280×800 (min 900×600), titleBarStyle: 'hidden'
      • titleBarOverlay dinámico por tema (colores en theme.store.ts)
      • electron-builder.json5 con extraResources comentado para empaquetar vendor/python-3.12 → python-runtime
   • contextIsolation: true, nodeIntegration: false
    • IPC handlers:
        win:minimize, win:maximize, win:close, win:isMaximized
        titlebar:set-overlay, app:getVersion, shell:open-external
        chat:send (DELEGA AL SIDECAR PYTHON vía stdio JSON-RPC)
        chat:abort
        sidecar:status (health check del proceso Python)
        security:validateMessage, security:sanitizeToolCall, security:checkRateLimit
        security:sanitizeToolCalls, security:status (Rust native addon)
        keymanager:push, keymanager:pushAll, keymanager:clear (vault keys → Python)
        memory:list, memory:addEntry, memory:updateEntry,
          memory:deleteEntry, memory:addRelation, memory:updateRelation,
          memory:removeRelation
        vault:isAvailable, vault:storeKey, vault:getKey,
          vault:deleteKey, vault:listKeys, vault:hasKey
         search:web (Brave Search — DEPRECADO @deprecated, migrar a web_search tool del sidecar Python)
         terminal:create, terminal:write, terminal:resize, terminal:destroy,
           terminal:agent-write (PTY real node-pty)
     • Exposición via contextBridge:
        window.electronAPI → min/max/close/isMaximized/onMaximizedChange/
                             setTitleBarOverlay/getVersion
        window.electron → IPC genérico (on/send/invoke)
        window.sparta → onEvent/sendEvent + sendMessage(vendor,providerId,mode,
                         skills,mcpServers,semanticMemory,reasoning)
                        + abortMessage + isSidecarReady
         window.vault → isAvailable/storeKey/getKey/deleteKey/listKeys/hasKey
         window.terminal → create/write/resize/destroy/onData/onExit/agentWrite

 4.7 Routing:

   No hay router. La navegación es por estado de MainView (ui.store.ts):
     MainView =
       | { type: 'chat'; sessionId?: string }
       | { type: 'editor' }
       | { type: 'terminal' }
       | { type: 'agents' }
       | { type: 'sessions' }
       | { type: 'skills' }
       | { type: 'mcp' }
       | { type: 'channels'; channelId?: string }
       | { type: 'memory' }

   ui.store.ts también guarda previousMainView y provee goBack().

 18. FLUJO DE EVENTOS THINKING (doc flujo-thinking-events.txt)
================================================================================

 18.1 Visión general

   El flujo de "thinking" representa el razonamiento interno del modelo de IA
   antes de generar su respuesta visible. Cuando un modelo soporta "thinking"
   (Claude extended thinking, DeepSeek-R1, QwQ, etc.), emite tokens de
   razonamiento antes de los tokens de contenido.

   Eventos clave:
     • thinking:started   → El modelo comenzó a razonar
     • thinking:token     → Un token de razonamiento (stream)
     • thinking:completed → El razonamiento terminó

   Estos eventos viajan Python → Transporte → Frontend de forma independiente
   a los eventos stream:token (contenido visible).

 18.2 Backend (Python / LangGraph)

   Archivo: python/sparta_ai/streaming/event_bridge.py

   LangGraph ejecuta graph.astream_events() y cada evento LangGraph se
   despacha mediante _dispatch_event() o _dispatch_event_ws().

   Fuentes de thinking tokens:
     A) reasoning_content en metadata (OpenAI-compatible)
        ──────────────────────────────────────────────────
        Proveedores como DeepSeek / OpenRouter / Qwen ponen el razonamiento
        en additional_kwargs.reasoning_content o response_metadata.reasoning.
        _extract_reasoning_content(chunk) extrae ese string y emite
        thinking:started + thinking:token.

     B) Content blocks tipo "thinking"/"reasoning" (Anthropic)
        ──────────────────────────────────────────────────
        Claude devuelve content blocks con type="thinking".
        _is_reasoning_block() lo detecta; _block_text() extrae el texto.

     C) additional_kwargs.thinking en on_chat_model_start
        ──────────────────────────────────────────────────
        Algunos proveedores indican que van a pensar mediante
        additional_kwargs.thinking al inicio del modelo.
        Se emite thinking:started inmediatamente.

   Secuencia de eventos:
     on_chat_model_stream:
       ├── reasoning_content? → thinking:started (1ra vez) + thinking:token
       ├── content blocks?
       │   ├── type=thinking/reasoning → thinking:started + token
       │   └── type=text              → stream:token
       └── content string             → stream:token

     on_chat_model_start:
       └── additional_kwargs.thinking → thinking:started

     on_chat_model_end:
       ├── usage → evento "usage"
       └── thinking_active? → thinking:completed (con tokens_used)

     on_chain_end (name="agent"):
       ├── thinking_active? → thinking:completed (fallback)
       └── stream:completed

   Estado mutable (stream_state):
     stream_state = {"thinking_active": False}
     - Se pone True la primera vez que se emite thinking:started.
     - Evita emitir múltiples thinking:started.
     - Se vuelve False en thinking:completed.
     - Es independiente para cada request_id.

   Dos transportes:
     • Electron (stdio): stream_agent_to_electron()
       → _emit(request_id, "thinking:token", {...})
       → Escribe JSON a stdout: {"id":"...","event":"...","data":{...}}

     • Web (WebSocket): stream_agent_to_websocket()
       → _emit_ws_renderer(ws, "thinking:token", {...})
       → Envía JSON por WS: {"type":"thinking:token", "sessionId":..., ...}

 18.3 Capa de transporte

   Electron: sidecar.ipc.ts + chat.ipc.ts
     Python stdout
       {"id":"sid:mid","event":"thinking:token",
        "data":{"token":"Let me think..."}}
       ↓
     sidecar.ipc.ts
       Lee línea del stdout del subproceso Python
       Emite sidecarEvents.emit('sidecar:message', msg)
       ↓
     chat.ipc.ts (handler onMessage)
       switch (event):
         'thinking:started'  → sendToRenderer({sessionId, messageId,
                                                type:'thinking:started'})
         'thinking:token'    → sendToRenderer({..., type:'thinking:token',
                                                token, chunkSeq})
         'thinking:completed'→ sendToRenderer({..., type:'thinking:completed',
                                                tokensUsed})
       ↓
     win.webContents.send('sparta:event', event)
       ↓
     preload.ts → window.sparta.onEvent(callback)

     Nota: chunkSeq es un contador secuencial (getNextSeq) para detectar
     chunks duplicados del lado del renderer.

   Web: WebSocket directo
     Python FastAPI (server_web.py) → WebSocket /ws
       _emit_ws_renderer() manda msg:
       {"type":"thinking:token","token":"...",
        "sessionId":"...","messageId":"..."}
       ↓
     Browser WebSocket onmessage
       JSON.parse → SpartaEvent
       ↓
     WebAdapter.onEvent handlers

 18.4 Frontend (Renderer / React)

   messaging-adapter.ts:
     Unifica ambos transportes bajo la misma interfaz:
       • ElectronAdapter: usa window.sparta.onEvent/onMessage/sendMessage
       • WebAdapter: usa WebSocket

     export const messagingAdapter = IS_ELECTRON
       ? new ElectronAdapter()
       : new WebAdapter()

     onEvent(handler: (event: SpartaEvent) => void): () => void
       → Registra handler para eventos entrantes

   useChatSession.ts:
     useEffect:
       messagingAdapter.onEvent((event: SpartaEvent) => {
         switch (event.type) {
           case 'thinking:started':
             store.onThinkingStart(sid, mid)
             store.updateMessage(mid, { reasoningStartedAt: Date.now() })
             break
           case 'thinking:token':
             store.appendThinking(sid, mid, token, chunkSeq)
             break
           case 'thinking:completed':
             store.onThinkingEnd(sid, mid, tokensUsed)
             store.updateMessage(mid, { reasoningCompletedAt: Date.now() })
             break
           case 'stream:token':
             store.appendContent(sid, mid, token, chunkSeq)
             break
           case 'stream:completed':
             store.onStreamEnd(sid, mid)
             stopStreamingFn(sid)
             // → post-procesamiento: extracción de memoria
             break
         }
       })

   chat.store.ts (Zustand):
     Estado del mensaje (Message type):
       {
         reasoningText?: string           // texto de razonamiento acumulado
         thinkingStatus?: ThinkingStatus  // 'idle'|'streaming'|'completed'|'collapsed'
         thinkingTokensUsed?: number      // tokens usados en razonamiento
         reasoningStartedAt?: number      // timestamp inicio
         reasoningCompletedAt?: number    // timestamp fin
         lastThinkChunkSeq?: number       // para dedup de chunks
       }

     onThinkingStart(sessionId, messageId):
       → message.thinkingStatus = 'streaming'

     appendThinking(sessionId, messageId, delta, chunkSeq):
       → Si chunkSeq <= lastThinkChunkSeq → descarta (dedup)
       → message.reasoningText += delta
       → message.thinkingStatus = 'streaming'
       → message.isStreaming = true
       → message.lastThinkChunkSeq = chunkSeq

     onThinkingEnd(sessionId, messageId, tokensUsed):
       → message.thinkingStatus = 'completed'
       → message.thinkingTokensUsed = tokensUsed

     onStreamEnd(sessionId, messageId):
       → message.isStreaming = false

   Renderizado en UI (MessageBubble.tsx):
     hasThinking = message.thinkingStatus !== 'idle'
       ↓
     <ThinkingBlock>
       ├── reasoningText mostrado colapsable
       ├── thinkingStatus controla animación (streaming = pulse)
       └── thinkingTokensUsed mostrado al final

 18.6 Manejo de duplicados (dedup)

   Tanto content como thinking tokens llevan un chunkSeq numerado
   secuencialmente desde el lado Electron (chat.ipc.ts → getNextSeq()).

   En el store:
     • content:  lastChunkSeq
     • thinking: lastThinkChunkSeq

   Si llega un chunk con chunkSeq <= lastChunkSeq, se descarta con un
   console.warn. Esto protege contra eventos duplicados por race condition.

   En WebSocket no hay chunkSeq (el WS es ordenado y confiable por
   naturaleza), pero el store maneja el campo si estuviera presente.
