 1. PROBLEMAS DE THINKING / RAZONAMIENTO (resueltos)
================================================================================

 El flujo de thinking va: Python → IPC → useChatSession → chat.store → MessageBubble → ThinkingBlock

 1.1 Bug A — appendThinking no transicionaba 'starting' → 'streaming'
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/stores/chat.store.ts
    Síntoma: thinkingStatus se quedaba pegado en 'starting' para siempre.
             El cursor parpadeante nunca aparecía (solo se muestra si
             status === 'streaming').
    Causa:  appendThinking() solo revisaba:
              msg.thinkingStatus === 'idle' || msg.thinkingStatus === undefined
            Pero onThinkingStart() pone el status en 'starting', no en
            'idle'. La transición nunca ocurría.
    Fix:    Agregado 'starting' a la condición:
              msg.thinkingStatus === 'idle' || msg.thinkingStatus === 'starting'
              || msg.thinkingStatus === undefined

 1.2 Bug B — Código muerto: onThinkingToken
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/stores/chat.store.ts
    Síntoma: Existía una función onThinkingToken que SÍ tenía el fix correcto
             (con 'starting'), pero NADIE la llamaba. El código real usaba
             appendThinking() que NO tenía el fix.
    Causa:  Dos implementaciones del mismo concepto, solo una activa.
    Fix:    Eliminado onThinkingToken de la interfaz ChatState y su
             implementación. Todo pasa por appendThinking() ahora.

 1.3 Bug C — hasThinking no incluía 'starting'
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/components/chat/MessageBubble.tsx
    Síntoma: Al recibir thinking:started, el frontend no montaba el
             <ThinkingBlock>. El usuario veía la burbuja vacía sin ningún
             indicador durante segundos (hasta llegar el primer token).
    Causa:  hasThinking solo revisaba:
              'streaming' | 'completed' | 'collapsed' | reasoningText > 0
            Pero no 'starting'.
    Fix:    Agregado 'starting' a la condición.

 1.4 Bug D — Auto-colapso del ThinkingBlock
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/components/chat/reasoning/ThinkingBlock.tsx
    Síntoma: El bloque de thinking se cerraba solo a los 1.2s de completado.
             El usuario perdía la oportunidad de ver el razonamiento.
    Causa:  useEffect que escuchaba status === 'completed' && hasResponseContent
             y cerraba isExpanded con setTimeout.
    Fix:    Eliminado el useEffect de auto-colapso y la prop hasResponseContent
             (ya no se necesita porque nunca se auto-colapsa).

 1.5 Bug — hasProvider no revisaba hasVaultKey
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/components/input/ChatInput.tsx
    Síntoma: En web mode, al configurar un proveedor cloud, apiKey quedaba
             en undefined (la key se guarda cifrada en localStorage via
             web-vault). El check hasProvider solo revisaba:
               p.kind === 'local' || p.apiKey
             Como apiKey es undefined, hasProvider siempre era false.
             El input se bloqueaba y mostraba "No hay proveedores con API
             key configurados" aunque el usuario SÍ había configurado uno.
    Fix:    Agregado p.hasVaultKey al some():
               providers.some((p) => p.kind === 'local' || p.apiKey || p.hasVaultKey)

 1.6 Bug — WebAdapter enviaba camelCase a Python (snake_case)
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/lib/messaging-adapter.ts
    Síntoma: En web mode, el WebAdapter.sendMessage() enviaba el objeto
             ChatSendRequest directamente con campos camelCase:
               providerKey, mcpServers, semanticMemory
             Pero Python server_web.py espera:
               provider_key, mcp_servers, semantic_memory
             En particular provider_key llegaba como null, y Python
             retornaba inmediatamente "No provider key provided".
    Fix:    Mapeo explícito de campos camelCase → snake_case en sendMessage().

 1.7 Bug — messages[] incluía el assistant vacío (Cohere error 400)
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/hooks/useChatSession.ts
    Síntoma: Cohere retornaba error 400:
               "missing required parameter: 'messages[2].content'"
             El mensaje assistant se crea con content: '' como placeholder
             para el streaming, pero se incluía en el array messages enviado
             al provider. Cohere (y otros) rechazan mensajes sin contenido.
    Fix:    Filtrar el assistant message por ID antes de armar el array:
               .filter((m) => m.id !== assistantId)

 1.8 Bug — TerminalPanel.tsx lanzaba throw a nivel de módulo
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/components/terminal/TerminalPanel.tsx
    Síntoma: En web mode, al importar TerminalPanel, se ejecutaba:
               if (!FEATURES.terminal) throw new Error(...)
             El throw a nivel de módulo crasheaba el import y potencialmente
             impedía que otras partes de la app funcionaran.
    Fix:    Movido el guard a inside de la función componente:
               export function TerminalPanel() {
                 if (!FEATURES.terminal) return null
                 ...
               }

 1.9 Bug — Username hardcodeado en ChatInput placeholder
 ────────────────────────────────────────────────────────────────────────────
    Archivo: src/components/input/ChatInput.tsx
    Síntoma: El placeholder decía "Ask anything..." en vez de la traducción
             correspondiente (español/inglés).
    Fix:    (Pendiente — usar sistema i18n)

 3. LO QUE DEBE MOSTRAR EL FRONTEND (y estados actuales)
================================================================================

 3.1 Bloque de Thinking (ThinkingBlock)
 ────────────────────────────────────────────────────────────────────────────
    Estado: ✅ FUNCIONAL (con modelos que emiten reasoning_content)
    Flujo:
      Python emite thinking:started
        → frontend: onThinkingStart → thinkingStatus = 'starting'
        → MessageBubble: hasThinking = true (incluye 'starting')
        → ThinkingBlock: muestra <ThinkingSkeletonRows> (3 líneas animadas)
      Python emite thinking:token (N veces)
        → frontend: appendThinking → thinkingStatus = 'streaming'
        → ThinkingBlock: muestra líneas con iconos + cursor parpadeante
        → ThinkingPill: muestra timer "X.Xs" + spinner
      Python emite thinking:completed
        → frontend: onThinkingEnd → thinkingStatus = 'completed'
        → ThinkingBlock: muestra check + "Pensó por X.Xs" + tokens usados
        → NO se auto-colapsa (el usuario decide)

    Requisito para verlo: modelo que emita reasoning_content.
    ✅ DeepSeek R1 y destilados
    ✅ Anthropic Claude (extended thinking)
    ❌ Cohere, Llama, Mistral, GPT (no emiten)

 3.2 ModelPicker (selector de modelos)
 ────────────────────────────────────────────────────────────────────────────
    Estado: ✅ FUNCIONAL (con providers que tienen models[] poblado)
    Requisito: el usuario debe hacer "Test Connection" en ConfigureProviderDialog
               para que los modelos se guarden en provider.models[].

    Pendiente: fetch automático al abrir el combobox (sin requerir test manual).
