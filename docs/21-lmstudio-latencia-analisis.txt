1. RESUMEN EJECUTIVO
================================================================================

Cuando usas LM Studio con un modelo de razonamiento (thinking), la respuesta
en Sparta Agent es significativamente mas lenta que en el chat nativo de LM
Studio por CINCO razones principales:

  a) El pipeline de Sparta tiene 7+ capas de procesamiento intermedio
  b) Los modelos thinking generan razonamiento largo ANTES de la respuesta
  c) El sistema injecta instrucciones de razonamiento emulado
  d) Cada token pasa por think_scrubber, repetition_guard, y skill_detector
  e) La compresion de contexto puede invocar al LLM para resumir historial

El chat nativo de LM Studio es directo: UI -> HTTP -> LLM -> Stream -> UI.
Sparta Agent es: React -> IPC/WS -> Python -> LangGraph -> Provider -> LLM
                  -> Stream -> EventBridge -> ThinkScrubber -> RepetitionGuard
                  -> IPC/WS -> React -> Zustand -> RAF Throttle -> DOM

3. CUELLOS DE BOTTLENEQUE IDENTIFICADOS (POR FASE)
================================================================================

CRITICO (impacto > 1s):
─────────────────────────

[1] LLM inference con thinking/reasoning (FASE 4a)
    Ubicacion: sparta_agent.py:305-313 (llm_with_tools.ainvoke)
    Problema: Cuando reasoning_enabled=True, el modelo genera tokens de
    razonamiento ANTES de la respuesta visible. Un modelo con thinking
    puede generar 2000-8000 tokens de razonamiento que el usuario NO ve
    hasta que termina todo el bloque de thinking.
    
    En LM Studio especificamente:
    - Los modelos Qwen2.5, DeepSeek-R1, etc. generan tags <think>
    - El modelo NO empieza a producir respuesta visible hasta que
      termina de razonar internamente
    - En LM Studio chat, el thinking se muestra en tiempo real
    - En Sparta, el think_scrubber lo procesa pero el frontend espera
      el cierre del bloque thinking antes de mostrar la respuesta
    
    Impacto: 5-30 segundos de "espera silenciosa" para el usuario

[2] System prompt excesivamente largo (FASE 4a)
    Ubicacion: sparta_agent.py:115-216
    Problema: El system prompt contiene:
    - Instrucciones del agente (~500 tokens)
    - Contexto del proyecto (_build_project_context)
    - Reglas operativas (~300 tokens)
    - Instrucciones de razonamiento emulado (~200 tokens extra)
    - Skills activos (variable)
    - Memoria contextual (variable)
    Total estimado: 1500-4000 tokens solo de system prompt
    
    Con un modelo local, estos tokens se procesan en PREFILL.
    LM Studio procesa todo el prompt antes de generar el primer token.
    
    Impacto: 0.5-3 segundos de prefill

[3] Context compression puede invocar al LLM (FASE 3)
    Ubicacion: context_manager.py:16-53, server_handlers.py:431
    Problema: Si hay >20 mensajes Y >40,000 caracteres, se genera un
    resumen usando el LLM. Esto es una llamada ADICIONAL al modelo
    antes de empezar a generar la respuesta.
    
    Impacto: 2-10 segundos extra cuando se activa

[4] Emulated reasoning prompt injection (FASE 4a)
    Ubicacion: emulated_reasoning.py:32-57, sparta_agent.py:222-228
    Problema: Para modelos en LM Studio (que son "local"), se detecta
    que necesitan razonamiento emulado (needs_emulated_reasoning=True).
    Se agrega al system prompt una instraccion de ~200 tokens que le
    pide al modelo que envuelva su razonamiento en tags <think>.
    
    Esto es COUNTERPRODUCTIVO porque:
    - El modelo YA tiene thinking interno (es un modelo reasoning)
    - La instruccion le pide que piense DOS veces
    - Los tags generados deben ser parseados por ThinkScrubber
    - Se duplica el trabajo de razonamiento
    
    Impacto: 2-10 segundos de razonamiento redundante

MODERADO (impacto 100ms-1s):
─────────────────────────────

[5] message_cleanup con deepcopy (FASE 3)
    Ubicacion: message_cleanup.py:46, 129, 258
    Problema: Tres llamadas a copy.deepcopy() en secuencia:
    - drop_thinking_only_and_merge_users() -> deepcopy
    - copy_reasoning_content_for_api() -> deepcopy
    - format_reasoning_for_provider() -> deepcopy
    Cada deepcopy de una lista de 20+ mensajes con tool_calls es costoso.
    
    Impacto: 10-100ms por conversacion larga

[6] Tool binding (FASE 3)
    Ubicacion: sparta_agent.py:81-86
    Problema: Se crean 3 variantes del LLM con herramientas bindeadas:
    - llm_with_tools (todos los tools + delegate tools)
    - llm_plan (solo read-only tools + delegate tools)
    - llm_chat (chat tools + research delegate)
    Cada bind_tools serializa los JSON schemas de todas las herramientas.
    Con 15+ herramientas, esto agrega tokens al prompt.
    
    Impacto: 50-200ms

[7] Skill detection por token (FASE 5)
    Ubicacion: chat_model_stream.py:65, 90, 146
    Problema: En CADA token de reasoning, se ejecuta detect_skill()
    que compara contra patrones regex. Con miles de tokens de thinking,
    esto acumula overhead.
    
    Impacto: 5-20ms total

[8] RAF throttle en frontend (FASE 7)
    Ubicacion: useStreamEvents.ts:83-89
    Problema: requestAnimationFrame limita updates a 60fps. Los tokens
    se bufferizan hasta el proximo frame. Esto agrega 0-16ms de latencia
    por batch de tokens.
    
    Impacto: 0-16ms (negligible pero acumulativo)

[9] stdout flush batching (FASE 5)
    Ubicacion: event_bridge.py:24, 58-66
    Problema: _FLUSH_BATCH_SIZE = 8. Se hace flush cada 8 tokens.
    Esto agrega buffering que puede causar micro-pausas.
    
    Impacto: 1-5ms por batch

BAJO (impacto < 100ms):
───────────────────────

[10] JSON serialization por token (FASE 5)
     Ubicacion: event_bridge.py:57
     Problema: Cada token se serializa a JSON individualmente.
     json.dumps() por token es ineficiente.
     
     Impacto: 1-3ms por 100 tokens

[11] RepetitionGuard buffer growth (FASE 5)
     Ubicacion: repetition_guard.py:58-83
     Problema: El buffer crece indefinidamente. Con respuestas largas,
     _check() opera sobre buffers de miles de caracteres.
     
     Impacto: 1-5ms por check

[12] WebSocket reconnection (Web mode only)
     Ubicacion: messaging-adapter.ts:103-106
     Problema: Si el WS se desconecta, reconecta cada 2s.
     Durante la reconexion, no hay streaming.
     
     Impacto: Variable (solo en desconexiones)

4. EL PROBLEMA DE LOS MODELOS THINKING/REASONING EN LM STUDIO
================================================================================

4.1 COMO FUNCIONA THINKING EN LM STUDIO
─────────────────────────────────────────

LM Studio expone modelos con thinking/reasoning via la API OpenAI-compatible
en http://localhost:1234/v1. Cuando cargas un modelo como DeepSeek-R1,
Qwen2.5-Coder-32B, o cualquier modelo con capacidad de reasoning:

  - El modelo genera tokens de razonamiento internos
  - LM Studio los expone en un campo separado (reasoning_content)
  - O los envuelve en tags <think>...</think> en el content

4.2 QUE HACE SPARTA CON THINKING
──────────────────────────────────

Para LM Studio (vendor="lmstudio"), el codigo en providers.py:109-127
NO configura reasoning en extra_body (solo lo hace para openrouter y deepseek).

Pero en emulated_reasoning.py:73-96, needs_emulated_reasoning() retorna
True para lmstudio porque no esta en la lista de vendors con reasoning nativo.

Esto significa que Sparta INYECTA al system prompt una instruccion que le
pide al modelo que genere tags <think> manualmente:

  sparta_agent.py:222-228:
    if needs_emulated_reasoning(vendor, model):
        system = append_reasoning_prompt(system)

  emulated_reasoning.py:32-57:
    _REASONING_INSTRUCTION = """
    Antes de responder, razoná internamente sobre la tarea...
    Usá EXCLUSIVAMENTE estos tags: <think> ... </think>
    """

4.3 EL PROBLEMA ESPECIFICO
───────────────────────────

Si el modelo YA es un modelo de reasoning (DeepSeek-R1, Qwen-QwQ, etc.):
  1. El modelo YA genera thinking interno nativamente
  2. Sparta le pide que TAMBIEN use tags <think> artificiales
  3. El modelo produce DOS capas de razonamiento:
     - reasoning_content nativo del modelo
     - Tags <think> en el content (por la instruccion)
  4. El ThinkScrubber debe parsear AMBOS
  5. Se duplica el trabajo de razonamiento = 2x tiempo

Si el modelo NO es reasoning (Llama 3.2, Qwen sin R, etc.):
  1. La instruccion emulated es correcta y necesaria
  2. El modelo genera tags <think> en el content
  3. ThinkScrubber los separa y emite como thinking:token
  4. Esto funciona bien pero agrega overhead de parsing

4.4 FLUJO DE UN TOKEN CON THINKING EN SPARTA
─────────────────────────────────────────────

  LM Studio genera chunk con reasoning_content:
    │
    ▼
  _extract_reasoning_content() [chat_model_stream.py:52]
    │
    ▼
  ¿Tiene reasoning_content? ──SI──► thinking_started event
    │                                    │
    │                                    ▼
    │                              thinking:token event
    │                                    │
    │                                    ▼
    │                              _detect_and_emit_skill()
    │                              (regex sobre cada token de reasoning)
    │
    NO (es content string)
    │
    ▼
  StreamingThinkScrubber.feed(content) [think_scrubber.py:37]
    │
    ▼
  ¿Contiene tags <think>? ──SI──► Separar reasoning vs visible
    │                              │
    │                              ▼
    │                        reasoning -> thinking:token
    │                        visible -> stream:token
    │
    NO
    │
    ▼
  RepetitionGuard.feed(visible) [repetition_guard.py:58]
    │
    ▼
  ¿Repeticion detectada? ──SI──► stream:degenerate (abort)
    │
    NO
    │
    ▼
  _emit() -> stdout -> IPC -> React -> RAF throttle -> DOM

  Cada paso agrega latencia. Con miles de tokens, el overhead total
  puede ser de 100-500ms solo en el pipeline de procesamiento.

5. COMPARATIVA: LM STUDIO CHAT vs SPARTA AGENT
================================================================================

LM STUDIO CHAT (nativo):
  Usuario escribe → HTTP POST /v1/chat/completions → LLM inference
  → Stream SSE chunks → UI renderiza inline
  Capas: 3 (HTTP, LLM, UI)
  Latencia percepcionada: ~200ms primer token, streaming inmediato
  Thinking: se muestra en tiempo real en panel separado

SPARTA AGENT (con LM Studio):
  Usuario escribe → Zustand store → messagingAdapter → IPC/WS
  → Python prepare_agent (3x deepcopy, compresion, skill load)
  → build_sparta_graph (bind_tools x3, compilar grafo)
  → LangGraph astream_events → agent_node (system prompt 4K tokens)
  → LLM ainvoke → ThinkScrubber → RepetitionGuard → skill_detector
  → _emit JSON → stdout flush → IPC → React event handler
  → _queueContent → RAF throttle → Zustand appendContent
  → React re-render
  Capas: 12+
  Latencia percepcionada: 1-5s primer token, streaming con overhead
  Thinking: procesado pero con overhead de parsing

DIFERENCIA: ~10-25x mas capas en Sparta

7. SOLUCIONES RECOMENDADAS
================================================================================

SOLUCION 1 [ALTA PRIORIDAD]: Deshabilitar emulated reasoning para LM Studio
  Problema: needs_emulated_reasoning() no detecta modelos reasoning
  Solucion: En emulated_reasoning.py:73-96, agregar un check para
  modelos que LM Studio reporta como reasoning via /v1/models.
  
  Alternativa rapida: En sparta_agent.py:222-228, NO inyectar el
  reasoning prompt cuando el modelo ya tiene thinking nativo.
  
  Impacto estimado: -30% a -50% tiempo de respuesta

SOLUCION 2 [ALTA PRIORIDAD]: Reducir system prompt
  Problema: 1500-4000 tokens de system prompt
  Solucion: Comprimir el system prompt a ~800 tokens maximo.
  Mover reglas detalladas a un documento que el modelo consulte.
  
  Impacto estimado: -20% tiempo de prefill

SOLUCION 3 [MEDIA PRIORIDAD]: Eliminar deepcopy innecesario
  Problema: 3x deepcopy en message_cleanup
  Solucion: Operar in-place o usar shallow copy donde sea seguro.
  
  Impacto estimado: -50ms por request

SOLUCION 4 [MEDIA PRIORIDAD]: Reducir flush batch size
  Problema: _FLUSH_BATCH_SIZE = 8 causa buffering
  Solucion: Cambiar a 1 o 2 para streaming mas fluido.
  
  Impacto estimado: -10ms latencia percibida

SOLUCION 5 [BAJA PRIORIDAD]: Optimizar skill detection
  Problema: Regex ejecutado por cada token de reasoning
  Solucion: Ejecutar skill detection cada N tokens en vez de cada uno.
  
  Impacto estimado: -5ms total

SOLUCION 6 [ALTA PRIORIDAD]: Paralelizar prepare_agent
  Problema: prepare_agent() ejecuta compresion, skills, memory, MCP
  secuencialmente cuando podrian correr en paralelo.
  Solucion: Usar asyncio.gather() para las tareas independientes.
  
  Impacto estimado: -100-500ms

SOLUCION 7 [MEDIA PRIORIDAD]: Cache de tool schemas
  Problema: bind_tools() serializa JSON schemas en cada request
  Solucion: Cachear los schemas si las herramientas no cambiaron.
  
  Impacto estimado: -50-100ms

El overhead total de Sparta sobre LM Studio nativo es de ~3-7 segundos,
principalmente debido a:
  1. Thinking emulado duplicado (+2-5s)
  2. System prompt mas largo (+0.5-2s prefill)
  3. Pipeline de procesamiento (+0.1-0.3s)
  4. Preparacion del agente (+0.2-0.5s)

NOTA: Estos tiempos son ESTIMADOS y varian segun:
  - GPU disponible (VRAM, velocidad de inferencia)
  - Tamano del modelo (7B vs 32B vs 70B)
  - Longitud del historial de conversacion
  - Numero de herramientas bindeadas
  - Configuracion de LM Studio (n_threads, GPU offload, etc.)
