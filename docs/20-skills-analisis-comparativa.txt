2. LO QUE SPARTA YA TIENE BIEN
================================================================================

✅ Progressive disclosure (2 niveles):
   - Nivel 1: <available_skills> en system prompt (lightweight index)
   - Nivel 2: skill_view_tool() para cargar SKILL.md completo
   
✅ Skill detection en runtime:
   - detect_skill_in_thought() durante streaming
   - Emite skill:activated event a la UI

✅ Skill toggle on/off (Zustand store persistido)

✅ Carga desde Vite glob import + Python filesystem scan

✅ Security guard básico con scan_skill_content()

================================================================================
3. LO QUE SPARTA DEBE MEJORAR (priorizado)
================================================================================

✅ COMPLETADO (v0.3.0):

A) SKILLS COMO USER MESSAGES
   Implementado en sparta_agent.py: agent_node inyecta active skills como
   role:user, no en system prompt. Sistema prompt estable = prefix cache ok.

B) SKILL_MANAGE TOOL (CRUD)
   Implementado en skill_tools.py: skill_manage_tool con create/edit/patch/delete/list.
   Escribe SKILL.md en ~/.sparta/skills/<category>/<id>/. Limpia cache automáticamente.

C) SKILL_LIST TOOL
   Implementado en skill_tools.py: skills_list_tool(category?) devuelve skills
   agrupadas por categoría con descripciones e indicadores featured.

D) SECURITY SCANNER
   Expandido de 8 → 78 patrones en 11 categorías (exec, exfiltration, ransomware,
   jailbreak, crypto mining, fork bomb, etc). Trust levels, risk scoring.

E) BACKGROUND REVIEW
   Implementado en background_review.py: revisión autónoma cada 10 tool calls,
   analiza transcripción, sugiere creación/parcheo de skills via LLM.

F) PLATFORM GATING
   Implementado en skill_loader.py: skills con platforms: [macos, linux, windows]
   se filtran según OS actual. Skills sin platform field = compatibles siempre.

G) SKILL BUNDLES
   Implementado en skill_bundles.py: YAML en ~/.sparta/skill-bundles/*.yaml.
   create_bundle(), get_bundle(), build_bundle_context() para carga múltiple.

H) CURATOR
   Implementado en curator.py: active → stale (30d) → archived (90d).
   run_curator(), restore_skill(), list_archived(). Solo user-created skills.
   Nunca skills builtin, hub-installed o protegidas. Nunca borra, solo archiva.

I) USAGE TELEMETRY
   Implementado en skill_usage.py: bump_view(), bump_use(), bump_patch().
   Persistido en ~/.sparta/skills/.skill_usage.json. get_stale_skills(), get_most_used().

J) PREPROCESSING
   Implementado en skill_preprocessing.py: ${HERMES_SKILL_DIR}, ${HERMES_SESSION_ID},
   ${HERMES_OS}, ${HERMES_USER_NAME}. Inline shell !`command` (OFF por seguridad).
   Configurable via SPARTA_SKILL_TEMPLATE_VARS y SPARTA_SKILL_INLINE_SHELL.

K) SLASH COMMAND /skill-name
   Implementado en SlashCommandMenu.tsx + ChatInput.tsx: /<skill-id> activa skill,
   /skill lista todas, /skill <id> view muestra detalle.

L) INDEX CACHE
   @lru_cache(maxsize=1) en skills_index(). clear_skill_cache() pública.

4. COMPARATIVA DETALLADA: PROGRESSIVE DISCLOSURE
================================================================================

HERMES:
  Nivel 1: <available_skills> en system prompt (solo nombres + desc 60 chars)
           Cacheado: LRU in-process + disk snapshot
  Nivel 2: skill_view(name) → SKILL.md completo como user message
           (NO en system prompt = no invalida prefix cache)
  Nivel 3: skill_view(name, file_path) → linked files
  Compact mode: categorías no-coding muestran solo nombres (sin desc)

SPARTA:
  Nivel 1: <available_skills> en system prompt (id + name + desc completa)
           Cacheado: @lru_cache simple (solo en proceso)
  Nivel 2: skill_view_tool(id) → SKILL.md como tool output
           (El output va al CONTEXTO del LLM, no es user message)
  No hay Nivel 3 ni compact mode

================================================================================
5. COMPARATIVA DETALLADA: ACTIVACIÓN Y DETECCIÓN
================================================================================

HERMES (3 formas de activación):
  1. Automática: el agente VE el índice en system prompt y decide usar skill_view
  2. Por tool call: skills_list() + skill_view() para cargar contenido
  3. Por slash command: /skill-name → build_skill_invocation_message() →
     inyecta como user message con scaffolding [IMPORTANT: skill activada...]
  4. CLI preloading: --skills "s1,s2" → inyecta en mensaje inicial

SPARTA (2 formas de activación):
  1. Manual: usuario hace toggle on/off en UI
  2. Por tool call: el agente ve <available_skills> en system prompt y puede
     llamar skill_view_tool()
  No hay slash command /skill-name
  No hay preloading

================================================================================
6. COMPARATIVA DETALLADA: INYECCIÓN EN CONTEXTO
================================================================================

HERMES:
  • Índice (<available_skills>) → system prompt (stable tier, prefix-cached)
  • Contenido completo → user message (NO toca system prompt)
  • Preloaded → user message inicial
  • Background review → fork separado

  BENEFICIO: El system prompt NUNCA cambia por skills activas.
             El prefix cache del proveedor NO se invalida.
             Solo el índice compacto está en system prompt.

SPARTA:
  • Índice (<available_skills>) → system prompt (variable)
  • Active skills (<active_skills>) → system prompt (variable)
  • Contenido completo → tool output (LangChain tool)

  PROBLEMA: system prompt cambia según skills activas.
            El prefix cache se invalida en CADA request.
            Cada toggle de skill = miss de cache.

8. RESUMEN EJECUTIVO
================================================================================

Sparta Agent tiene una base sólida de skills con progressive disclosure y
skill detection en runtime. Sin embargo, carece de características clave:

  1. 🚨 Las skills se inyectan en system prompt (invalida prefix cache)
     → Hermes usa user messages para preservar cache

  2. 🚨 No hay skill_manage tool → el agente NO puede crear/editar skills
     → Hermes permite al agente auto-crear skills

  3. 🚨 No hay background review → el sistema no mejora solo
     → Hermes fork un AIAgent que parchea skills automáticamente

  4. ⚠️ Security scanner básico (10 patrones vs 70+)
  
  5. ⚠️ No hay platform/environment gating

  6. ⚠️ No hay bundles, curator, telemetría

Mejora más impactante con menor esfuerzo:
  Implementar skill_manage tool + mover contenido a user messages.
  Esto desbloquea que el agente cree sus propias skills y preserva cache.
